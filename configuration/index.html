用最小配置建立可理解、可回退、可审计的 Codex 工作环境。

决策工作台 配置总览

这块把普通阅读页改成一次可执行判断：先看入口和材料，再记录证据、失败处理和验收动作。

入口 本地配置 + 只读验证任务
材料 当前目录、默认权限、审批策略
证据 配置差异、只读任务记录、回退说明
完成信号 最小配置可运行、可解释、可回退
检查点动作材料/证据完成判断
输入判断先判断本页任务是否匹配当前材料、入口和风险等级。当前目录、默认权限、审批策略材料不匹配时先缩小任务，不进入执行。
过程证据执行前写清禁止动作，执行后留下可打开的检查材料。配置差异、只读任务记录、回退说明另一个人能看懂判断链路。
失败处理遇到停止条件时记录现场，不扩大权限、不补猜缺失信息。配置会默认扩大目录或网络访问停止原因和下一步负责人明确。
验收动作用本页给出的命令、截图、清单或人工检查证明完成。最小配置可运行、可解释、可回退通过项、待确认项和回退方式都已记录。

可复用判断单

目标：建立当前项目的最小可用配置
限制：不保存密钥，不默认扩大工作区
验证：让 Codex 只读说明会访问哪些文件
交付：配置 diff、验证记录、回退步骤

配置目标

配置不是为了打开所有能力，而是为了让 Codex 的默认行为可预测、可解释、可回退。每项设置都应能回答：它解决什么问题，增加什么风险，如何验证。

  • 先从最小可用配置开始。
  • 把默认工作区限制在当前项目。
  • 把高风险动作转为显式审批。

标准操作流程

每次调整配置后，都应使用一个低风险任务验证行为是否符合预期。不要在生产项目中第一次测试新配置。

  • 记录修改前后的配置差异。
  • 运行一个只读检查任务，确认 Codex 能解释会访问哪些文件。
  • 运行一个小范围写入任务，确认 diff 可控且可回退。
  • 把成功配置写入团队文档。

验收标准

专业配置的验收不是“能运行”，而是“能解释、能限制、能复盘”。

  • Codex 能说明当前任务需要哪些文件、命令或网络访问。
  • 不相关目录不会被默认读取或修改。
  • 高风险操作有明确审批点。
  • 配置变更有记录并可回退。

文档质量验收标准

本页只有在读者能按步骤执行、识别风险、产出交付物并完成验收时才算达标。

  • 中文和英文信息一致，没有遗漏关键限制。
  • 读者能根据本页完成一个可回退的低风险任务。
  • 任务结束后有输出、证据和待确认事项。
  • 涉及动态事实时，读者能回到官方文档复核。

Reusable Decision Brief

Goal: create the minimum usable configuration for the current project
Constraints: do not save secrets or broaden the workspace by default
Verification: ask Codex to explain which files it would access read-only
Deliverable: configuration diff, validation log, rollback steps

Configuration objective

Configuration is not about enabling every capability. It is about making Codex behavior predictable, explainable, and reversible. Every setting should answer: what problem does it solve, what risk does it add, and how will it be verified?

  • Start with the minimum viable configuration.
  • Limit the default workspace to the current project.
  • Turn high-risk actions into explicit approvals.

Standard operating procedure

After every configuration change, validate behavior with a low-risk task. Do not test a new configuration for the first time in a production project.

  • Record the before-and-after configuration diff.
  • Run a read-only inspection task and confirm Codex can explain which files it will access.
  • Run a small write task and confirm the diff is bounded and reversible.
  • Document the successful configuration in team documentation.

Acceptance criteria

Professional configuration is not accepted because it runs; it is accepted because it can be explained, constrained, and reviewed.

  • Codex can explain which files, commands, or network access the current task needs.
  • Unrelated directories are not read or modified by default.
  • High-risk operations have explicit approval points.
  • Configuration changes are recorded and reversible.

Documentation Quality Acceptance Criteria

This page meets the documentation standard only when readers can follow the procedure, identify risks, produce deliverables, and complete acceptance checks.

  • Chinese and English content are equivalent and do not omit critical constraints.
  • A reader can complete a reversible low-risk task using this page.
  • The task ends with output, evidence, and confirmation items.
  • For dynamic facts, the reader can verify against official documentation.
