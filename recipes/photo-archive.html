05 Markdown 知识库重整

把散乱 Markdown 笔记整理成统一字段、目录索引和可搜索主题清单。

文档状态
发布草案 v0.2
最后核对
2026-05-28
适用读者
业务人员、创作者、项目维护者
预计耗时
35-60 分钟
风险等级
低到中

任务背景

把散乱 Markdown 笔记整理成统一字段、目录索引和可搜索主题清单。

  • 场景：知识管理。
  • 入口：本地文件 + Markdown 检查。
  • 证据：字段清单、迁移 diff、索引页、缺口表。

环境与材料

先把可用环境、允许读取的材料和禁止动作写清楚，再让 Codex 开始处理。

  • 复制一份笔记目录作为工作副本，避免直接改原始资料。
  • 选定统一字段：标题、日期、主题、状态、摘要、行动项。
  • 先处理 5 篇样本，确认规则后再批量迁移。

操作剧本

把任务拆成可观察的步骤，要求 Codex 先说明处理方式，再产出文件、表格或报告。

  • 先让 Codex 扫描文件名和标题，输出字段缺失矩阵。
  • 把迁移规则写成清单：哪些字段可自动推断，哪些必须人工补。
  • 批量修改后生成索引页，按主题和状态分组。

关键步骤

每一步都要留下中间结果，避免只得到一份无法复查的最终文本。

  • 抽样读取笔记，识别命名混乱、标题重复、日期缺失和标签不一致。
  • 生成 frontmatter 模板和迁移计划，先展示 diff 再写入。
  • 创建 index.md，列出主题、状态、摘要和下一步。

过程证据

实战案例的价值不在于说做完了，而在于能展示做了什么、怎么判断、哪里需要人确认。

  • 本案例的核心证据是：字段清单、迁移 diff、索引页、缺口表。
  • 证据文件和截图应使用稳定命名，方便回看和比对。
  • 不确定项必须集中记录，不夹在正文里被忽略。

结果样例

输出必须能被另一个人独立检查，不能只是一段聊天记录。

  • 每篇笔记都有统一字段，缺失内容标为待补，不编造。
  • 目录索引可按主题快速找到材料，也能看到过期或未完成项。
  • 迁移 diff 可复查，必要时能回退单篇文件。

踩坑记录

以下问题在真实任务中最容易导致返工，应在任务单里提前写明。

  • 不要让 Codex 猜具体日期、作者或结论；缺失就标注待确认。
  • 一次性处理全部文件风险较高，先用样本校准规则。
  • 索引页不是目录树复制，应突出状态和下一步。

风险边界

Codex 可以协助分析、生成和检查，但账号动作、公开发布和高风险改动必须由用户确认。

  • 确认后批量迁移，并生成主题索引和待补清单。
  • 保留原始材料或原始文件，改动版使用新文件名。
  • 涉及金额、日期、权限、对外承诺和个人信息时，必须人工复核。

可复用任务单

把下面三句复制成任务起点，再替换成自己的材料、约束和验收方式。

  • 请扫描这个 Markdown 目录，先输出字段缺失矩阵和迁移规则。
  • 请先处理 5 个样本文件，展示 diff 后再等待确认。
  • 确认后批量迁移，并生成主题索引和待补清单。

文档质量验收标准

本页只有在读者能按步骤执行、识别风险、产出交付物并完成验收时才算达标。

  • 中文和英文信息一致，没有遗漏关键限制。
  • 读者能根据本页完成一个可回退的低风险任务。
  • 任务结束后有输出、证据和待确认事项。
  • 涉及动态事实时，读者能回到官方文档复核。

05 Markdown Knowledge Base Restructure

Restructure scattered Markdown notes into consistent fields, an index, and searchable topic list.

Status
Release draft v0.2
Last verified
2026-05-28
Audience
Operators, creators, and project maintainers
Estimated time
35-60 minutes
Risk level
Low to medium

Task Background

Restructure scattered Markdown notes into consistent fields, an index, and searchable topic list.

  • Scenario: Knowledge management.
  • Entry: Local files plus Markdown checks.
  • Evidence: Field inventory, migration diff, index page, and gap table.

Environment and Materials

Define the available environment, allowed materials, and prohibited actions before Codex starts processing.

  • Duplicate the notes directory as a working copy before editing original material.
  • Choose consistent fields: title, date, topic, status, summary, and action items.
  • Process five samples first and batch-migrate only after rules are confirmed.

Operating Script

Split the task into observable steps and ask Codex to state the handling method before producing files, tables, or reports.

  • Ask Codex to scan filenames and headings, then output a missing-field matrix.
  • Write migration rules as a checklist: which fields can be inferred and which require manual input.
  • After batch edits, generate an index grouped by topic and status.

Key Steps

Every step should leave an intermediate result instead of only producing final text that cannot be reviewed.

  • Sample notes to detect naming drift, duplicate headings, missing dates, and inconsistent tags.
  • Generate the frontmatter template and migration plan, then show diffs before writing.
  • Create index.md with topic, status, summary, and next action.

Evidence Trail

The value of a practical recipe is not saying it is done, but showing what was done, how it was judged, and what needs human confirmation.

  • The core evidence for this recipe is: Field inventory, migration diff, index page, and gap table.
  • Evidence files and screenshots should use stable names for review and comparison.
  • Uncertainties must be centralized instead of being buried in body text.

Result Sample

The output must be independently reviewable by another person, not just a chat transcript.

  • Every note has consistent fields; missing content is marked pending, not invented.
  • The index supports quick topic lookup and highlights stale or incomplete items.
  • Migration diffs are reviewable and individual files can be reverted if needed.

Pitfalls

The following issues most often cause rework in real tasks and should be stated in the work order up front.

  • Do not let Codex guess dates, authors, or conclusions; mark missing fields for confirmation.
  • Processing all files at once is risky; calibrate rules on samples first.
  • The index is not a copied folder tree; it should emphasize status and next actions.

Risk Boundaries

Codex can assist with analysis, generation, and checking, but account actions, publication, and high-risk changes require user confirmation.

  • After confirmation, batch-migrate and generate a topic index plus pending list.
  • Keep original materials or files and use a new filename for edited versions.
  • Amounts, dates, permissions, external promises, and personal information must be manually reviewed.

Reusable Work Order

Use the following three lines as the starting work order, then replace them with your own materials, constraints, and acceptance method.

  • Scan this Markdown directory and first output a missing-field matrix and migration rules.
  • Process five sample files first, show the diff, and wait for confirmation.
  • After confirmation, batch-migrate and generate a topic index plus pending list.

Documentation Quality Acceptance Criteria

This page meets the documentation standard only when readers can follow the procedure, identify risks, produce deliverables, and complete acceptance checks.

  • Chinese and English content are equivalent and do not omit critical constraints.
  • A reader can complete a reversible low-risk task using this page.
  • The task ends with output, evidence, and confirmation items.
  • For dynamic facts, the reader can verify against official documentation.
