05 Markdown 知识库重整

把散乱 Markdown 笔记整理成统一 frontmatter、主题索引和待补清单，并保留样本 diff 便于回退。

文档状态
发布草案 v0.2
最后核对
2026-05-28
适用读者
研究者、创作者、个人知识库维护者
预计耗时
50-80 分钟
风险等级
低到中
入口 本地 Markdown 目录 + 只读扫描
材料 笔记副本、字段规则、样本文件、目标索引格式
证据 字段缺失矩阵、样本 diff、索引页、待补清单
交付物 重整后的 Markdown 副本、index.md、pending-review.md
风险 低到中
预计耗时 50-80 分钟

任务背景

把散乱 Markdown 笔记整理成统一 frontmatter、主题索引和待补清单，并保留样本 diff 便于回退。

知识管理 研究者、创作者、个人知识库维护者 低到中

输入材料

本次任务先锁定可读取材料、禁止动作和输出格式，再开始生成或检查。

  • 只处理笔记目录副本，原始目录保持只读。
  • 统一字段为 title、date、topic、status、summary、next_action。
  • 缺失字段标记为待补，不猜作者、日期或结论。
notes-working/
  2025-ideas.md       # 无 frontmatter，标题重复
  meeting-ai.md       # 有标题，无日期
  draft-launch.md     # 状态不清楚
目标字段: title, date, topic, status, summary, next_action

运行环境

  • 样本目录：notes-working/，先抽 5 篇验证规则。
  • 索引按 topic 和 status 分组，不复制完整目录树。
  • 所有批量修改先展示 diff，再等待确认。

操作剧本

  1. 第一步只扫描文件名、标题和已有 frontmatter，输出缺失矩阵。
  2. 第二步处理 5 篇样本并展示 diff，确认规则后再批量迁移。
  3. 第三步生成主题索引和待补清单，突出状态和下一步。

过程证据

下面的表格记录本次任务如何判断完成，而不是只描述最终结果。

结果样例

结果样例保留可复核字段、文件名和待确认项，便于另一个人接手检查。

index.md
## 产品想法
- Team Notes / active / 下一步：补充使用场景
- Launch Draft / pending / 下一步：确认发布日期

pending-review.md
- meeting-ai.md: date 缺失
- draft-launch.md: status 需要确认

失败与修正

失败点现象修正动作
自动猜日期文件名年份被误当成完整日期只保留可证明日期，其他标待补
索引太像目录初版只列文件名，无法行动改为 topic、status、next_action 三列
批量范围过大一次改全部文件难以审阅先做样本 diff 再批量

风险边界

  • 原始笔记目录不写入，迁移只发生在工作副本。
  • 缺失信息标记为 pending，不补写没有证据的内容。
  • 批量前必须展示样本 diff 并确认规则。

验收标准

  • 每篇笔记都有统一字段或明确待补标记。
  • index.md 能按主题和状态快速定位材料。
  • pending-review.md 集中列出需要人工补充的信息。
  • 样本 diff 可读，单篇文件可回退。
验收命令与人工步骤 rg --files notes-working | sortrg -n "^title:|^date:|^topic:|^status:" notes-workinggit diff -- notes-working index.md pending-review.md

文档质量验收标准

本页只有在读者能按步骤执行、识别风险、产出交付物并完成验收时才算达标。

  • 中文和英文信息一致，没有遗漏关键限制。
  • 读者能根据本页完成一个可回退的低风险任务。
  • 任务结束后有输出、证据和待确认事项。
  • 涉及动态事实时，读者能回到官方文档复核。

Restructure scattered Markdown notes into consistent frontmatter, a topic index, and a pending list while keeping sample diffs for rollback.

Task Background

Restructure scattered Markdown notes into consistent frontmatter, a topic index, and a pending list while keeping sample diffs for rollback.

Knowledge management Researchers, creators, and personal knowledge-base maintainers Low to medium

Input Materials

This task first locks readable materials, prohibited actions, and output format before generation or inspection.

  • Only the copied notes folder is processed; the original folder stays read-only.
  • Consistent fields are title, date, topic, status, summary, and next_action.
  • Missing fields are marked pending; author, date, and conclusion are not guessed.
notes-working/
  2025-ideas.md       # no frontmatter, duplicate heading
  meeting-ai.md       # has title, no date
  draft-launch.md     # unclear status
Target fields: title, date, topic, status, summary, next_action

Run Environment

  • Sample folder: notes-working/; validate rules with five notes first.
  • The index groups by topic and status instead of copying the full folder tree.
  • All batch edits show diffs before confirmation.

Operating Script

  1. Step one scans only filenames, headings, and existing frontmatter, then outputs a missing-field matrix.
  2. Step two processes five samples and shows diffs; batch migration happens only after rules are confirmed.
  3. Step three generates topic index and pending list, emphasizing status and next action.

Failures and Corrections

FailureSymptomCorrection
Guessed datesFilename year was treated as full dateKeep only provable dates and mark others pending

Batch too largeEditing all files at once was hard to reviewCreate sample diffs before batching

Risk Boundaries

  • The original notes folder is not written; migration happens only in the working copy.
  • Missing information is marked pending and absent content is not invented.
  • Before batching, sample diffs and rules must be confirmed.

Acceptance Criteria

  • Every note has consistent fields or explicit pending markers.
  • index.md supports quick lookup by topic and status.
  • pending-review.md centralizes information needing human input.
  • Sample diffs are readable and individual files can be reverted.
