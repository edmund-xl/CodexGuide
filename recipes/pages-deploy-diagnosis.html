03 GitHub Pages 部署失败诊断

从失败通知、Actions job 和本地检查命令定位 Pages 发布问题，并用新的 run 与线上响应完成闭环。

文档状态
发布草案 v0.2
最后核对
2026-05-28
适用读者
项目维护者、内容站负责人
预计耗时
40-70 分钟
风险等级
中
入口 本地仓库 + Actions 日志
材料 失败截图、run id、workflow 文件、最新提交 SHA
证据 失败步骤、修复提交、新 run、线上 200 响应
交付物 deploy-incident.md、修复提交、验证命令记录
风险 中
预计耗时 40-70 分钟

任务背景

从失败通知、Actions job 和本地检查命令定位 Pages 发布问题，并用新的 run 与线上响应完成闭环。

发布排障 项目维护者、内容站负责人 中

输入材料

本次任务先锁定可读取材料、禁止动作和输出格式，再开始生成或检查。

  • 失败页面截图只用于识别 job 名称和失败状态，不作为唯一判断。
  • 本地工作区先确认干净，避免发布修复混入内容改动。
  • 只改 workflow 或 Pages 配置相关文件，不碰业务页面。
失败摘要
Run: Deploy Pages #5
build: failed in 10 seconds
deploy: skipped
现象: 邮件显示 Some jobs were not successful
限制: 不改内容页面，先判断 workflow、artifact、Pages 设置哪一段失败

运行环境

  • 仓库分支：main，发布方式：GitHub Pages Actions。
  • 本地质量门禁：npm run build、npm run check、git diff --check。
  • 线上验收：目标地址返回 200 OK，页面包含本次标题或新内容标记。

操作剧本

  1. 先区分内容构建失败、artifact 上传失败、Pages 配置失败和部署失败。
  2. 进入 job steps 找第一个红色步骤，不被最终 skipped 状态误导。
  3. 修复后用新 run 验证，旧失败 run 不会自动变绿。

过程证据

下面的表格记录本次任务如何判断完成，而不是只描述最终结果。

检查点观察结果证据文件状态

结果样例

结果样例保留可复核字段、文件名和待确认项，便于另一个人接手检查。

deploy-incident.md
现象：Deploy Pages 的 build 失败，deploy 跳过
根因：workflow 的 Pages 配置步骤未满足当前发布方式
修复：更新 Pages Actions 配置并保留 build/check 步骤
验证：新 run build=success, deploy=success; live URL=200 OK

失败与修正

失败点现象修正动作
只看邮件邮件没有第一个失败步骤必须打开 job steps
旧 run 误判修复后旧 run 仍显示失败以新提交触发的新 run 为准
缓存延迟线上页面短时间仍显示旧内容用响应头和页面标题共同判断

风险边界

  • 部署修复与内容改动分开提交，便于回退。
  • 不使用强制覆盖远端历史，除非用户单独确认。
  • 每次推送后等待新 run 完成，再宣布线上成功。

验收标准

  • 本地 build 和 check 均通过。
  • 失败记录写清现象、失败步骤、根因、修复和验证命令。
  • 新 run 的 build 与 deploy 均为 success。
  • 线上地址返回 200，并能看到新页面内容。

03 GitHub Pages Deployment Failure Diagnosis

Diagnose a Pages deployment issue from failure notification, Actions jobs, and local checks, then close the loop with a new run and live response.

Task Background

Diagnose a Pages deployment issue from failure notification, Actions jobs, and local checks, then close the loop with a new run and live response.

Deployment troubleshooting Project maintainers and documentation site owners Medium

Input Materials

This task first locks readable materials, prohibited actions, and output format before generation or inspection.

  • The failure screenshot is used only to identify job name and status, not as the only judgment.
  • Confirm a clean local worktree first so deploy fixes do not mix with content edits.
  • Change only workflow or Pages configuration-related files, not content pages.
Failure summary
Run: Deploy Pages #5
build: failed in 10 seconds
deploy: skipped
Symptom: email says Some jobs were not successful
Constraint: do not edit content pages; first identify whether workflow, artifact, or Pages settings failed

Run Environment

  • Repository branch: main; publishing method: GitHub Pages Actions.
  • Local quality gates: npm run build, npm run check, and git diff --check.
  • Live acceptance: target URL returns 200 OK and contains the current title or new content marker.

Operating Script

  1. First separate content build failure, artifact upload failure, Pages configuration failure, and deploy failure.
  2. Open job steps and find the first red step instead of being misled by the final skipped status.
  3. After fixing, validate with a new run; old failed runs do not turn green automatically.

Risk Boundaries

  • Keep deploy fixes and content changes in separate commits for rollback.
  • Do not force-overwrite remote history unless the user explicitly confirms.
  • After each push, wait for the new run to finish before declaring live success.

Acceptance Criteria

  • Local build and check both pass.
  • Incident record states symptom, failed step, root cause, fix, and verification commands.
  • The new run has build and deploy both successful.
  • Live URL returns 200 and displays the new page content.
